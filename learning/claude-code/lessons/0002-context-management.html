上下文管理：200K token 的秘密

Lesson 002 · Claude Code · 2026-06-15

Part of your mission to 理解 Claude Code 的核心架构和工具体系，能够自主设计、调试和优化自己的 skill 工作流。

学完这节课，你将能够 解释为什么 Claude Code 有时会"忘记"之前的信息，以及如何通过 CLAUDE.md 确保关键信息不丢失。

你一定遇到过的问题

假设你在写小说，对话已经进行了 2 小时：

你：记住，主角叫林默，她有一只蓝色眼睛。
Claude：好的，记住了。
...（聊了 50 轮）
你：林默的眼睛是什么颜色？
Claude：蓝色的。
你：对了，她小时候在哪里长大？
Claude：（胡说了一个地方，和你之前设定的完全不同）

为什么会这样？不是 Claude 变笨了，是上下文窗口装不下了。

上下文窗口：那张桌子

把上下文窗口想象成一张桌子：

  • 你和 Claude 的所有对话，包括你发的消息、Claude 的回复、工具读到的内容、工具执行的结果——全部都要摊在这张桌子上
  • Claude 每次"思考"的时候，需要看到桌上所有东西
  • 桌子有固定大小——Claude 的上下文窗口大约是 200K token（约 15 万中文字）

关键洞察：工具返回的内容会进入上下文

这是 Session 2 最重要的一课。当你让 Claude Code 读取一个文件或执行一个命令时：

你：读一下 novel/chapter-01.md
        ↓
Claude Code 调用 Read 工具
        ↓
Read 工具读取文件，返回 2000 行内容
        ↓
这 2000 行内容作为【工具返回结果】
        ↓
【放进上下文窗口】   ← 关键！
        ↓
Claude 大模型看到这些内容，开始思考

工具返回的每一行内容都会进入上下文窗口。 这正是 Claude Code 能理解你代码的原因——它把文件内容"喂"给大模型看。

Bash 命令也一样：stdout 的输出作为工具返回结果，放进上下文窗口。

信息溢出：桌子满了怎么办？

关键洞察：桌子上的东西不是永久的。当东西太多放不下时，最早的那些会被"推出桌面"——Claude 就"看不见"了。

这就像桌子满了之后，你把最早摆上去的东西推到地上——不是它们消失了，而是你看不见它们了。

Lost in the Middle：中间遗失

即使内容还在上下文窗口里，大模型对不同位置的内容关注度也不同：

这就是为什么：

  • 你的第 1 条消息（设定主角叫林默）在开头，通常能记住
  • 但第 30 条消息里你补充的一个重要约束，可能就在中间被忽略了
  • 你最后说的一句话又在结尾，所以往往能记住

解决方案：关键信息放哪里？

上下文窗口满了之后，最重要的信息不应该依赖对话历史，而应该放在不会被丢弃的地方：

方法 是否有效 原理 问题
CLAUDE.md ✅ 最佳 每次对话自动注入，永远在开头 无
记忆工具 ✅ 有效 持久存储，需要时主动读取 需要 Claude 主动决定去读
每次重复 ✅ 有效 新开对话时放在开头 太累，不可持续

对于写小说这种场景——关键设定放在 CLAUDE.md 里是最稳的。

架构全景

检查你的理解

  1. 为什么和 Claude Code 聊久了它会"忘记"之前说的话？两个可能的原因是什么？
    答案

    1) 信息溢出：上下文窗口满了，最早的内容被主动丢弃。2) Lost in the Middle：即使内容还在上下文窗口里，中间部分也容易被忽略。

  2. 当 Claude Code 用 Read 工具读取一个 5000 行的文件时，这些内容会去哪里？
    答案

    Read 工具最多返回 2000 行，这些内容作为"工具返回结果"进入上下文窗口，大模型会看到这些内容。所以读大文件会快速消耗上下文空间。

  3. 为什么 CLAUDE.md 是存放关键设定的最佳位置？
    答案

    CLAUDE.md 的内容作为系统提示在每次对话开始时自动注入，永远在上下文窗口的开头，不会被丢弃，也不占对话空间。

  4. 假设你已经和 Claude Code 聊了 100 轮，上下文快满了。你告诉它"林默有恐高症"。哪种方法能确保它下次还记得？
    答案

    最可靠的方法是把这条设定写进 CLAUDE.md。下次新开对话时，它会自动注入到上下文开头。在当前对话中重复说也有效，但新开对话后又会面临同样的问题。
